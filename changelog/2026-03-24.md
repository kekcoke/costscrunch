# 2026-03-24 — WAF v2 Hardening: SQLi Rule, Path-Specific Rate Limiting, Access Logging

## [Infra-P1] — WAF v2 WebACL Enhancement (5 rules + logging)

### Added
- **AWSManagedRulesSQLiRuleSet** (priority 2): SQL injection detection managed rule group, inserted between CommonRuleSet and KnownBadInputsRuleSet.
- **Path-specific rate limiting** (priority 5): 200 requests/5-minute window per IP, scoped to `POST /receipts/upload` via `AndStatement` combining `ByteMatchStatement` on method (`POST`) and URI path (`/receipts/upload`, `EXACTLY` positional constraint). Prevents abuse of the receipt upload presigned-URL endpoint without affecting general API traffic.
- **WAF access logging**: Dedicated CloudWatch LogGroup (`/aws/wafv2/{prefix}-waf-logs`) with 90-day retention. Implemented as a separate `AWS::WAFv2::LoggingConfiguration` resource (CDK 2.241+ requirement — not a property on `CfnWebACL`). Authorization header redacted from logs to prevent Cognito token leakage.
- **WAF LogGroup**: 90-day retention, environment-aware removal policy (RETAIN in prod, DESTROY otherwise).

### Changed
- **Global rate limit**: Increased from 2,000 to 5,000 requests/5-minute window per IP to reduce false positives at scale.
- **Rule priorities**: Reordered to accommodate new SQLi rule — KnownBadInputs shifted to priority 3, global rate limit to 4, receipt upload rate limit to 5.
- **WebACL description**: Added descriptive text (`"CloudFront WAF: managed rules + rate limiting"`).

### Cost Impact
- +$1/month (additional rule: SQLi)
- +$5–20/month (WAF access logging ingestion, traffic-dependent)
- +$0.50–2/month (CloudWatch log storage at 90-day retention)
- **Total delta: ~$7–19/month** (from ~$8 to ~$15–27)

### Notes
- CloudFront-scoped WebACL logs must be delivered to **us-east-1**. If this stack is deployed to another region, the LogGroup must be moved to a separate us-east-1 stack and referenced cross-stack (noted in-code).
- `CfnLoggingConfiguration` is a standalone CloudFormation resource in CDK 2.241+ — it cannot be set as a property on `CfnWebACL`. This was discovered during implementation and test debugging.

## [AppSec-P2] — Secrets Management: SSM Parameter Store + Secrets Manager

### Added
- **SSM Parameter Store** for configuration values:
  - `/costscrunch-{env}/bedrock-model-id` — Bedrock Claude model ID
  - `/costscrunch-{env}/vite-app-url` — Frontend URL for CORS
  - `/costscrunch-{env}/from-email` — SES sender email
  - `/costscrunch-{env}/pinpoint-app-id` — AWS Pinpoint application ID

- **Secrets Manager** for sensitive notification config:
  - `costscrunch-{env}/notification-config` — JSON secret containing `fromEmail` + `pinpointAppId`

- **Lambda environment variables** now store parameter paths instead of actual values:
  - `SSM_BEDROCK_MODEL_ID` — SSM parameter path for Bedrock model
  - `SSM_VITE_APP_URL` — SSM parameter path for frontend URL
  - `SECRET_NOTIFICATION_ARN` — Secrets Manager ARN for notification config

- **IAM permissions** granted per least-privilege:
  - `ssm:GetParameter` for Bedrock model parameter
  - `secretsmanager:GetSecretValue` for notification secrets

- **CDK Security Aspect** (`SecretsGuardAspect`): Fails synthesis if sensitive values are added as plain-text Lambda environment variables. Blocks patterns including:
  - Keys matching `api_key`, `secret`, `password`, `token`, `credential`, `bearer`
  - AWS access key patterns (`AKIA...`)
  - Email addresses (`@costscrunch.com/io/dev`)

### Added
- **`SecretsAudit.test.ts`**: 8 tests asserting:
  1. No hardcoded email addresses in Lambda env vars
  2. No hardcoded sensitive values (API keys, passwords, tokens)
  3. No secrets exposed in CloudFormation Outputs
  4. Notification secrets stored in Secrets Manager
  5. Configuration stored in SSM Parameter Store
  6. Lambda functions use SSM/SECRET path references (not values)
  7. Notifications Lambda has Secrets Manager read access
  8. `FROM_EMAIL` and `PINPOINT_APP_ID` not stored as plain env vars

### Removed
- **Hardcoded secrets** from `sharedEnv` object:
  - `BEDROCK_MODEL_ID` (hardcoded fallback)
  - `VITE_APP_URL` (hardcoded fallback)
  - `FROM_EMAIL` (was hardcoded on `notificationsLambda`)
  - `PINPOINT_APP_ID` (was hardcoded on `notificationsLambda`)

### Security Impact
| Control | Before | After |
|---------|--------|-------|
| Secrets in env vars | ✅ Sensitive values exposed | ❌ Paths only |
| CloudFormation outputs | ✅ No secrets | ✅ No secrets |
| CDK synthesis guard | ❌ None | ✅ Blocks sensitive env vars |
| Runtime secret retrieval | N/A | ✅ SSM + Secrets Manager |

### Notes
- Lambdas must be updated to retrieve secrets at runtime via AWS SDK (`SSMClient`, `SecretsManagerClient`)
- SSM parameters support runtime updates without Lambda redeployment
- VPC endpoints for SSM and Secrets Manager already provisioned in stack
