## Duplicate Receipt Detection

### Features
- **Deterministic hash computation** (`computeReceiptHash`) from normalized (merchant + date + amount)
- **Fuzzy matching** with Levenshtein distance (threshold: 10 edits) and amount tolerance (±$0.50)
- **ReceiptHashIndex GSI** for fast duplicate lookups in DynamoDB
- **EventBridge integration** emitting `DuplicateReceiptDetected` events

### Backend Changes

**`backend/src/utils/receiptHash.ts`**
- `computeEntityHash()` — SHA-256 hash from pipe-separated parts
- `normalizeReceiptFields()` — lowercase merchant, date-only (YYYY-MM-DD), amount rounded to 2 decimals
- `computeReceiptHash()` — convenience wrapper for receipt duplicate detection

**`backend/src/utils/fuzzyMatch.ts`**
- `levenshtein()` — O(n*m) single-row DP algorithm
- `fuzzyMatchReceipt()` — compares receipts with configurable tolerances
- Returns `FuzzyMatchResult` with `isDuplicate`, `similarity` (exact/fuzzy/none), `merchantDistance`, `amountDifference`

**`backend/src/lambdas/sns-webhook/index.ts`**
- Added `checkForDuplicate()` — queries GSI, uses fuzzy matching
- Added `emitDuplicateDetected()` — emits `DuplicateReceiptDetected` EventBridge event
- Writes `receiptHash`, `gsi3pk`, `gsi3sk` to expense records
- Graceful handling for missing GSI in dev/test environments

**`infrastructure/stacks/CostsCrunchStack.ts`**
- Added `ReceiptHashIndex` GSI (PK: `receiptHash`, projection: ALL)

## Automated Post-Deploy Rollback

### Features
- **Health check gate** — Waits 30s for propagation, then polls `/health` endpoint
- **Automatic rollback** — Triggers on 3 failed health checks (10s retry interval)
- **Dual rollback strategies** — Uses CloudFormation `Previous` template or redeploys `HEAD~1`
- **Slack notifications** — Distinct success/failure messages with deployment SHA
- **Always runs** — Uses `if: always()` so it executes even on partial failures

### Workflow Changes

**`.github/workflows/rollback.yml`** (NEW)
- Reusable workflow triggered by `workflow_call`
- Inputs: `environment`, `stack_name`, `api_url`, `deployment_sha`, `cdk_command`
- Health check loop: 3 attempts, 10s interval, 30s timeout per attempt
- Rollback steps: `aws cloudformation rollback-stack` → `aws cloudformation wait stack-rollback-complete`
- Fallback: Git checkout HEAD~1 + CDK redeploy if no previous template

**`.github/workflows/deploy.yml`**
- Added `post-deploy-staging` job (needs: `deploy-staging`, if: `always()`)
- Added `post-deploy-prod` job (needs: `deploy-prod`, if: `always()`)
- Removed inline smoke test (now handled by rollback workflow)
- Both jobs call `rollback.yml` via `uses: ./.github/workflows/rollback.yml`

### Health Check Configuration

| Environment | API URL | Stack Name |
|-------------|---------|------------|
| staging | `${{ secrets.STAGING_URL }}/api` | `costscrunch-staging-CostsCrunchStack` |
| production | `https://api.costscrunch.io` | `costscrunch-prod-CostsCrunchStack` |

### Testing Rollback

1. Deploy to staging normally
2. Temporarily add `return 500` in health Lambda handler
3. Push a commit to trigger new deploy
4. Watch `post-deploy-staging` fail health checks and trigger rollback
5. Revert the 500 change and redeploy
