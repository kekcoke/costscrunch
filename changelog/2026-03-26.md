# Changelog — 2026-03-26

## 🛠 Infrastructure: Fixed Duplicate Lambda Containers in LocalStack (Opt 2)

### 🔴 Issue
Running the environment via `npm run dev:opt2` resulted in redundant Lambda containers for the same function name (specifically observed with `AnalyticsFunction`). Instead of a 1:1 mapping between function definitions and running sibling containers, multiple instances were spawned, leading to resource waste and potential state inconsistencies during local development.

### 🔍 Root Cause
The issue was identified as a race condition within LocalStack's `docker-reuse` executor during the intensive API Gateway registration phase:
- **Redundant Permissions**: The `bootstrap.sh` script previously called `lambda add-permission` for every individual route and HTTP method.
- **Triggered Refreshes**: Since `AnalyticsFunction` is registered against 4 distinct routes (`/analytics`, `/summary`, `/trends`, `/chart-data`), LocalStack received multiple configuration update requests in rapid succession.
- **Executor Race**: These overlapping requests triggered the `docker-reuse` executor to spawn new sibling containers ("refreshing" the function) before the previous one had stabilized, resulting in orphaned duplicate containers.

### ✅ Resolution
Refactored `infrastructure/localstack/opt2/bootstrap.sh` to decouple function permissions from route registration:
1.  **Atomic Deployment**: Moved `lambda add-permission` out of the per-route loop.
2.  **Global Authorization**: Added a single, global permission statement per function during the initial deployment phase using a wildcard `source-arn` (`arn:aws:execute-api:us-east-1:000000000000:*`).
3.  **Stability**: By "touching" the Lambda configuration only once per function, we ensure the executor stabilizes on a single container instance regardless of how many routes or methods are subsequently attached to it via API Gateway.
